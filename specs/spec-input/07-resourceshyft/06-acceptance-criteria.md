# Acceptance Criteria

## Service directory core

- organizations can be created and updated
- service offerings can be created and updated
- service locations can be linked appropriately

## Search and discovery

- users can search service offerings
- users can filter and refine results
- result ranking is useful enough for internal workflows
- service detail is understandable and complete enough for action

## Availability

- service availability can be represented and displayed
- users can understand what is open now and what is available later today
- availability display is human-friendly

## Requirements visibility

- users can see important fit / requirement guidance
- ResourceShyft can display requirement information without taking ownership of eligibility logic

## Workflow launches

- service detail can launch into downstream case or future referral/program flows
- launch seams remain stable for later integration

## Freshness

- service records can track how recently information was checked
- staff can understand confidence/freshness of service details

## Safety

- ResourceShyft does not take ownership of eligibility, document, referral, or finance logic
- public-facing release remains optional and does not constrain internal-first design
